Данная лабораторная работа продолжает повествование главы о [многофайловой сборке](https://github.com/UsamG1t/Methodics_of_LinuxAppDev/blob/master/Methodical_manual/02_Multifile/2.%20%D0%9C%D0%BD%D0%BE%D0%B3%D0%BE%D1%84%D0%B0%D0%B9%D0%BB%D0%BE%D0%B2%D0%B0%D1%8F%20%D1%81%D0%B1%D0%BE%D1%80%D0%BA%D0%B0.md).

---

При написании крупных программ многофайловость позволяет лучше ориентироваться в программе, разделять блоки кода и пространства имён. Для автоматической сборки программ, состоящих из множества файлов, используется утилита [make](https://www.gnu.org/software/make/manual/make.html).

При работе с крупными проектами часто используются [сборочные библиотеки](https://github.com/UsamG1t/Methodics_of_LinuxAppDev/blob/master/Methodical_manual/02_Multifile/2.%20%D0%9C%D0%BD%D0%BE%D0%B3%D0%BE%D1%84%D0%B0%D0%B9%D0%BB%D0%BE%D0%B2%D0%B0%D1%8F%20%D1%81%D0%B1%D0%BE%D1%80%D0%BA%D0%B0.md#%D1%81%D0%B1%D0%BE%D1%80%D0%BA%D0%B0-%D1%81%D0%BE%D0%B1%D1%81%D1%82%D0%B2%D0%B5%D0%BD%D0%BD%D1%8B%D1%85-%D0%B1%D0%B8%D0%B1%D0%BB%D0%B8%D0%BE%D1%82%D0%B5%D0%BA), при этом не только сторонние, но и собирающиеся из исходных файлов специально в проекте. 

Попробуем построить пакет, в котором соберём два исполняемых файла, каждый со своим типом библиотеки.

Программа состоит из четырёх файлов: основной код, код исполняемой функции, глобальная переменная-счётчик и заголовочный файл.

Опишем `Makefile` для проекта и рассмотрим подробнее его рецепты:

`builder`
```console
[builder@localhost Multilab-1.0]$ vim Makefile  
[builder@localhost Multilab-1.0]$ cat Makefile    

GENS = Multilab-* README-* libout.*  
TRASH = *.o *~ o.*  
CFLAGS = -Wall -fPIC  
CC = cc  
  
.PHONY: clean distclean  
.SECONDARY:     fun.o const.o  
.INTERMEDIATE:  libout.a(fun.o const.o)  
  
  
all:    binfiles documentation  
  
binfiles:       Multilab-a Multilab-so  
  
Multilab-a:     Multilab.o libout.a  
       $(CC) $(CFLAGS) $< -L. -lout -o $@  
  
Multilab-so:    Multilab.o libout.so  
       $(CC) $(CFLAGS) $^ -o $@  
  
libout.a:       libout.a(fun.o const.o)  
  
libout.so:      fun.o const.o  
       $(CC) $(CFLAGS) $^ -o $@ -shared  
  
  
fun.o Multilab.o:       outlib.h  
  
documentation:  README-a README-so
  
README-a:       Multilab-a  
       ./$< > $@ 2>&1  
  
README-so:      Multilab-so  
       LD_LIBRARY_PATH=`pwd` ./$< > $@ 2>&1  
  
clean:  
       rm -f $(TRASH)  
  
distclean:      clean  
       rm -f $(GENS)  
[builder@localhost Multilab-1.0]$  
```

 + `make` поддерживает множество встроенных рецептов, для работы которых используются переменные окружения. Это позволяет задавать рецепты, лишь описывая правильный формат цели и исходников;
 + При повторных сборках `make` ориентируется на метки времени создания и модификации файлов для проведения выборочной перекомпиляции. В случае самостоятельной генерации статических библиотек необходимы дополнительные цели для настройки правил пересборки;
 + Для каждого из целевых исполняемых файлов собирается своя библиотека на основе исходных файлов функции и глобальной переменной. Заметим, что для статических библиотек `make` поддерживает встроенный сценарий сборки, в то время как для динамической необходимо явно указывать сценарий;
 + При сборке конечного исполняемого файла в случае статической библиотеки необходимо указать её имя и месторасположение для сборки. В случае динамических библиотек указание обращения к библиотеке требуется только при исполнении;

Некоторые особенности данного `spec`-файла, связанные с библиотеками и `make`:
 + В директиве основной части `%build` используется не макрос автосборки `%make_build`, а явный вызов утилиты. Макрос запускает сборку с возможностью параллельного исполнения нескольких целей для ускорения, однако в данном проекте это приводит к нарушению зависимостей и падению сборки;
 + Расположение файлов при установке важно не только для самых исполняемых файлов, но и для **динамической** библиотеки. После получения исполняемого файла **статическая** библиотека **уже не будет нужна**;
